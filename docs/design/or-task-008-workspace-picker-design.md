# OR-TASK-008 工作区目录选择卡设计

更新时间：2026-03-18

## 背景

当前工作区切换主路径依赖 `/cwd <path>` 和 `/cd <path>`。这条路径有三个问题：

- 用户必须知道目录结构，且要手输路径，交互成本高。
- 产品入口仍然偏 CLI 心智，不像飞书里的用户产品。
- 目录切换语义和面板语义脱节，目录卡片只是 `/cwd` 的按钮外壳，不是真正的目录选择体验。

这轮要把“切目录”从命令式输入收敛为卡片式选择，把 `/cwd`、`/cd` 从用户主路径移除。

## 目标

- 提供一个面向飞书用户的工作区选择器，而不是命令提示。
- 以当前 release workspace 为边界，只暴露顶层目录，避免无限深路径浏览带来的复杂度。
- 让用户先选工作区，再选顶层目录，状态和后果都在同一张卡里说清楚。
- 卡片交互应支持原地刷新、分页和快捷入口复用。

## 不做什么

- 这轮不做任意深度的文件树浏览。
- 这轮不做搜索框或表单输入。
- 这轮不继续保留 `/cwd`、`/cd` 作为用户文档主路径。

## 交互模型

### 1. 主入口

- `/workspace` 打开工作区选择卡。
- `/panel workspace` 作为面板内的对应结果面。
- `/help` 和 `/panel` 里的相关按钮都跳转到 `/workspace`。

### 2. 卡片结构

工作区选择卡分成两段：

1. 工作区切换段
   - `main`
   - `develop`
   - 当前通道按钮置灰，避免重复点击。
2. 顶层目录段
   - 当前 workspace root 下的顶层目录列表
   - 当前目录高亮禁用
   - 当前目录所在的顶层目录用“当前所在顶层目录”态标识
   - 超出首屏时分页

### 3. 交互容器语义

采用 Feishu Card JSON 2.0 的 `interactive_container` 承载每一行目录入口：

- 整行可点击，不要求用户点小按钮。
- 行内同时承载标题、相对路径和当前态。
- callback value 统一回传命令和上下文。

参考文档：

- https://open.feishu.cn/document/feishu-cards/card-json-v2-components/containers/interactive-container

## 状态与边界

- 工作区根目录来自当前 release channel 对应的 workspace root。
- 目录选择只允许在该 root 内解析，不允许逃逸。
- 顶层目录列表默认过滤 `.` 开头目录，避免把 `.git`、`.venv` 等内部目录直接暴露给用户。
- 切换目录会清空当前 scope 的 backend thread 与上下文占用，下一条真实消息在新目录重新启动。

## 为什么只做顶层目录

这是刻意收敛，不是能力不足：

- 飞书卡片不适合承载深层文件树递归浏览。
- 对大多数项目，顶层目录已经足以表达“项目 / 模块 / 子系统”边界。
- 如果继续开放任意路径输入，产品又会退回 `/cwd` 的命令式心智。

后续如果确实需要更细粒度目录导航，应优先做“进入某个顶层目录后的二级目录卡片”，而不是恢复自由文本路径。

## 已落第一阶段实现

- 新增 `/workspace` 主入口。
- `/panel` 增加 `workspace` 结果面。
- 顶层目录列表从 `SessionWorkspaceService` 统一生成，支持分页。
- 快捷目录按钮改为复用 `/workspace select <path>`。
- `/cwd`、`/cd` 从帮助文案、README 和主面板语义中移除。

## 后续 follow-up

- 如果顶层目录很多，补“最近使用目录”或“固定目录”分组。
- 为工作区选择卡补统一的产品级视觉样式，而不是只停留在功能可用。
- 评估是否需要把 `/shortcut` 的维护也卡片化，避免用户继续记子命令。
