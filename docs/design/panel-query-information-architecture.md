# Panel Query & Information Architecture

## 目标

让飞书里的 `/panel` 从“会话面板”收敛为一个稳定的总入口：用户先进入总览，再按会话、目录、命令、状态四类高频对象切到对应结果面，用更短的路径完成恢复会话、切目录、查看现场和执行常用动作。

这轮要解决的不是“把所有能力都塞进一张卡”，而是先把最常见的入口关系理顺：

- 继续旧任务时，先看会话结果，而不是在 `/resume list` 和帮助文案之间来回切换。
- 进入某个目录时，先看目录结果里的稳定入口，而不是优先手写路径。
- 想做高频动作时，先看命令结果，而不是回忆完整命令集合。
- 不确定当前现场时，先看状态结果，再决定继续发消息、切目录还是切会话。

## 为什么 v1 先做“结构化筛选”，不先做自由文本搜索

`/panel` 所在的介质是飞书卡片，不是本地终端。

这里直接上自由文本搜索有两个问题：

- 输入成本更高：用户还是要切回命令或输入框，收益不如先把高频对象按结构化结果面收拢。
- 结果语义更难稳定：当前系统里真正稳定的主路径其实已经存在，分别是 `/resume`、`/cwd`、`/status`、`/help`；更应该先把“怎么更快进入这些主路径”收敛清楚，而不是先做一层模糊搜索。

因此这轮的决策是：

- `/panel` 承担“总入口 + 结果面导航”职责。
- 查询的 v1 形态优先采用**结构化筛选 / browse**，而不是自由文本搜索框。
- 每个结果面都只服务一种主目标，不混成一张大杂烩卡片。

后续如果会话量、目录量或命令量继续膨胀，再评估是否值得引入关键字搜索和结果排序。

## 查询对象范围

v1 明确覆盖四类高频对象：

1. **会话**：继续当前任务、恢复旧会话、按排序翻页浏览。
2. **目录**：进入常用目录、在 `main / develop` 之间切换工作位置。
3. **命令**：恢复上一条、新建会话、查看状态、打开帮助等高频动作。
4. **状态**：查看当前会话、上下文占用、原生会话和下一步建议。

这四类对象不是并列“功能列表”，而是四个结果面，各自对应一个用户意图。

## `/panel` 的职责边界

### `/panel` 负责什么

- 作为飞书里的**导航总入口**。
- 展示当前会话的核心上下文，避免用户切进结果面后失去方向。
- 提供稳定的结果面切换：`sessions / directories / commands / status`。
- 在结果面里用更短路径触发现有主命令。

### `/panel` 不负责什么

- 不替代 `/resume` 的命令语义；会话结果仍然复用 `/resume`。
- 不替代 `/cwd` 的目录切换语义；目录结果仍然复用 `/cwd`。
- 不替代 `/help` 的解释职责；帮助卡片仍负责“为什么这样做 / 下一步怎么发”。
- 不在这轮承担“全量配置面板”或“无限层级操作面板”职责。

一句话说，`/panel` 是入口和导航，不是第二套 runtime。

## 信息架构

### 1. 总览页

总览页保留三类信息：

- 当前会话摘要：会话 ID、通道、目录、模型、sandbox、上下文概览。
- 结果面入口：`会话 / 目录 / 命令 / 状态`。
- 预览区：最近会话与目录入口，用于直接扫描高频项。

总览页的目标不是“把所有动作都铺开”，而是帮助用户先决定“我现在要进入哪一类结果”。

### 2. 结果面

每个结果面保持相同骨架：

- 当前会话摘要
- 结果面切换导航
- 该类结果的简短说明
- 结果项列表
- 每个结果项只给一个主动作按钮

### 3. 结果项层级

所有结果项都复用同一套展示语义：

- **标题**：对象名或动作名
- **meta**：对象身份 / 当前状态 / 所属通道 / 目录等辅助信息
- **preview**：帮助用户做判断的短说明
- **主动作**：真正触发执行的按钮

这意味着：

- 总览页里的“最近会话”预览项和会话结果页里的项，语义一致。
- 总览页里的“目录入口”预览项和目录结果页里的项，语义一致。
- 结果面不会再各自长出一套不同的 item 结构。

## 组件映射决策

这轮明确收敛到以下映射：

- **会话项**：标题 + meta + preview + `/resume <token>` 主动作
- **目录项**：标题 + 路径 meta + 说明 + `/cwd <resolved-path>` 主动作
- **命令项**：动作标题 + 场景 meta + 说明 + 直接命令主动作
- **状态项**：状态标题 + 当前现场 meta + 说明 + `/status` 或 `/usage` 主动作

也就是说，查询结果与 `/panel` 主界面不是两套设计，而是同一套 item 语义在“预览态”和“结果态”上的复用。

## 与现有入口的关系

### `/resume list`

- 仍然保留，适合命令式使用和明确知道自己在找会话的人。
- `/panel sessions` 面向的是卡片扫描路径。
- 两者底层继续共享同一个 session browser 数据来源，而不是各自维护一份列表逻辑。

### 目录快捷入口

- 目录结果页只负责暴露入口，不重新定义切换语义。
- 真正切目录仍然复用 `/cwd`，这样状态流保持单一路径。

### `/help`

- `/help` 继续负责解释“现在该做什么”。
- `/panel commands` 负责把高频动作直接摆出来。
- 两者互补，而不是互相覆盖。

## 本轮最小落地范围

本轮落地四个结果面：

- `/panel`：总览页
- `/panel sessions`：会话结果页，支持 `--page` 与 `--sort`
- `/panel directories`：目录结果页
- `/panel commands`：命令结果页
- `/panel status`：状态结果页

其中会话结果页构成这轮的最小闭环：

1. 用户发送 `/panel`
2. 点击“会话”进入 `/panel sessions`
3. 在结果页翻页 / 排序 / 直接点按钮恢复会话
4. 实际恢复仍走 `/resume`

这条链路已经足以证明“结构化筛选优先”的路径能缩短会话恢复成本。

## 验收方式

自动化测试覆盖以下点：

- `/panel` 总览页会暴露四个结果面入口。
- `/panel sessions` 能展示会话结果，并保留排序、翻页和恢复动作。
- `/panel directories`、`/panel commands`、`/panel status` 都能发送对应结果卡片。
- `/panel` 命令解析支持 `sessions` 视图的 `--page` 与 `--sort`。

手工验收时可按下面路径走一遍：

1. 在飞书里发送 `/panel`，确认能看到 `会话 / 目录 / 命令 / 状态` 四个入口。
2. 点击“会话”，确认能在卡片里翻页、切排序，并点击某条会话恢复。
3. 点击“目录”，确认快捷目录按钮仍然复用 `/cwd`。
4. 点击“命令”与“状态”，确认结果项和总览页预览区的排版语义一致。

## 后续 follow-up

- 如果会话规模继续增大，评估是否为 `/panel sessions` 增加关键字搜索或游标分页。
- 如果目录入口需要用户自维护，评估是否增加收藏编辑或最近目录，而不是继续堆命令。
- 如果要做“终极 `/panel` 导航”，下一步重点是单卡翻页与返回路径，而不是先引入更多结果类型。
