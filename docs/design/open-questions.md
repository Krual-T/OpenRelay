# Open Questions

更新时间：2026-03-10

这份文档用于记录 `openrelay` 当前仍未收敛的设计问题，方便后续逐项讨论、拆解和实现。

## 1. 常用目录的快速切换

### 问题
当前虽然已经支持 `/cwd` 和 `/cd`，但在高频使用场景里，快速切换到几个常用目录仍然不够顺手。

### 需要讨论
- 是否要支持“常用目录 / 收藏目录 / 最近目录”
- 是否要支持目录别名，例如 `/cd work`、`/cd server`
- 是否要把常用目录放进 `/panel`，做成一键切换按钮
- 常用目录配置应该按“用户”“会话”还是“机器人实例”维度保存
- 常用目录和 `main / develop` 工作区切换之间应该如何配合

### 目标
让用户在飞书里切换到常用工作目录时，操作足够短、足够稳定、足够可预期。

## 2. 多个消息连续发送、编辑消息时的用户体验

### 问题
当用户在短时间内连续发送多条消息，或者在机器人回复过程中继续补充、修改、撤回信息时，当前体验还没有被明确设计清楚。

### 需要讨论
- 多条连续消息是应该进入队列串行处理，还是合并成一次 follow-up
- 如果前一条还在执行，后一条消息应该提示“排队中”、自动续写，还是询问用户是否中断前一条
- 用户编辑消息或撤回消息后，系统应该如何响应
- typing、流式卡片、状态提示在“排队 / 追加 / 覆盖 / 取消”几种场景下应该如何表现
- 群聊、私聊、线程回复这几种场景是否需要不同策略

### 目标
让用户在“连续输入”和“边想边补充”的真实使用习惯下，不会感到混乱，也不会误以为机器人漏看了消息。

## 3. 多机器人、多用户、多终端的支持方式

### 问题
未来可能不只是一台机器、一个机器人、一个人使用，而是会出现多机器人、多用户、多终端协作的需求。

### 需要讨论
- 是否支持一个 `openrelay` 进程管理多个机器人配置
- 是否支持不同用户各自拥有独立的机器人、独立的会话空间、独立的工作目录
- 如果同一个人有两台电脑，如何在不同终端之间灵活切换“当前会话实际跑在哪台机器上”
- 是否支持把同一个机器人会话重新 attach 到另一台机器 / 另一个终端
- 会话归属应该按“机器人”“飞书用户”“机器”“终端实例”还是“workspace”来建模
- 多实例同时在线时，如何避免两个终端争抢同一个会话
- 是否需要显式的“接管 / 迁移 / 释放 / 锁定”机制

### 目标
在保持会话隔离和可控性的前提下，让多机器人和多终端的使用方式足够清晰，不让用户搞不清楚“当前是谁在处理、在哪台机器上处理”。

## 4. 机器人回复过程中 `/stop` 无法停止

### 问题
当前观察到：机器人回复过程中发送 `/stop`，并不能稳定地停止当前回复。

### 需要讨论
- `/stop` 的中断请求是否被及时送达当前运行中的任务
- 当前锁模型和串行处理模型是否会把 `/stop` 卡在队列里
- backend 的取消语义是否真正实现到了 Codex / Claude 等执行层
- 停止后 typing、流式卡片、状态信息是否能够正确回收
- 停止成功后应该给用户什么样的确认反馈，以及是否保留部分中间输出

### 目标
让 `/stop` 成为一个可靠的“立即中断当前回复”动作，而不是一个不确定是否生效的指令。

## 后续建议

建议后续把这些问题逐项拆成独立 design note，每个问题至少补齐以下内容：
- 用户场景
- 期望行为
- 非目标
- 状态模型
- 命令与交互设计
- 实现边界
- 回滚与兼容策略
