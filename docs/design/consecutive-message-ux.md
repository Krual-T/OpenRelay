# 连续消息与 follow-up UX

更新时间：2026-03-10

## 背景

当前 `openrelay` 的 runtime 虽然实际上按 session 串行执行，但这个串行语义之前主要靠 `asyncio.Lock` 隐含成立：

- 用户连续发两条消息时，后一条只是在后台等锁，没有任何可见反馈。
- 用户会误以为机器人漏看了补充消息，或者以为必须等上一条完全结束后才能继续补充。
- `/stop` 能中断当前回复，但它和“已经补充过的下一条消息”之间没有清晰关系。
- 飞书入口目前只接了 `im.message.receive_v1` 和卡片 action，没有专门的“编辑 / 撤回”处理链路。

这会让“边想边补充”这条真实主路径显得不稳定，也让 active run 和串行处理模型停留在隐式约定层面。

## 目标

把 runtime 收敛成一条明确规则：

- 一个 session 同一时刻只跑一个 active run。
- 当前 run 期间到达的普通文本消息，不再静默等锁，而是进入明确的 follow-up 队列。
- 同一波连续补充的多条文本，按一轮 follow-up 合并，而不是让用户猜到底是排队还是漏看。
- `/stop` 只中断当前 run，不丢掉已经确认收到的 follow-up。
- 帮助文案和 README 明确告诉用户：什么时候直接补充，什么时候 `/stop`，什么时候 `/new`。

## 状态模型

每个 session 只维护三类状态：

1. **Idle**：没有 active run，也没有待处理输入。
2. **ActiveRun**：backend 正在处理当前 turn。
3. **PendingInputs**：当前 turn 之外，已经到达但尚未执行的后续输入。

其中 `PendingInputs` 再分两类：

- **QueuedFollowUp**：active run 期间到达的普通文本补充；如果连续到达，会在同一队列项里合并。
- **QueuedCommand / QueuedMessage**：其他排队输入，保持原到达顺序。

runtime 的执行方式也同步收敛：

- 如果当前 session 没有进行中的串行处理，消息直接进入主路径。
- 如果当前 session 正在处理，消息先进入 `PendingInputs`。
- 当前轮结束后，由同一条串行主路径继续 drain `PendingInputs`，直到 session 回到 Idle。

这样 active run、排队和 drain 的责任边界都是显式的，不再靠“谁刚好先拿到锁”来隐含表达产品语义。

## 用户可见规则

### 1. 连续发送 / 追加输入

- 当前回复还在生成时，用户继续发普通文本，不会被忽略。
- 第一条补充消息会立即收到确认：系统会在当前回复结束后自动继续。
- 同一波连续补充的后续文本会合并到下一轮 follow-up，而不是拆成多个零碎 turn。

这样做的目的很直接：

- 用户可以继续补充线索，不必停下来等。
- 系统给出明确确认，避免“它是不是没看到”的不确定感。
- 多条短补充会收敛成一轮更完整的下一步输入。

### 2. `/stop`

- `/stop` 只负责中断当前 active run。
- 如果当前 run 期间已经收到 follow-up，停止确认文案会显式说明“停止后会继续处理这些补充消息”。
- 当前 run 真正结束后，follow-up 仍按排队顺序继续执行。

这保证 `/stop` 的语义保持简单：

- 它不是“清空后续输入”；
- 它只是“立刻结束当前这一轮”。

如果用户是要彻底换话题，正确动作仍然是 `/new`。

### 3. 编辑 / 撤回

这轮不引入消息编辑状态机。

原因不是忽略这个场景，而是当前飞书接入只消费新消息事件和卡片 action：

- runtime 没有稳定的 edit / revoke 事件源；
- 如果现在硬加“猜测编辑意图”的兼容分支，只会把模型重新做模糊。

本轮明确策略是：

- **编辑上一条消息，不保证被 runtime 感知；**
- 想补充就直接再发一条；
- 想打断当前理解并重来，就先 `/stop`，再发修正后的内容；
- 想彻底切题，就 `/new`。

## 实现落点

- `src/openrelay/follow_up.py`：收敛 follow-up 合并与确认文案。
- `src/openrelay/runtime.py`：显式维护 pending session inputs，并在同一串行主路径里 drain。
- `src/openrelay/help_renderer.py`：补齐“当前回复还在跑时如何继续发消息”的帮助心智。
- `README.md`：补齐用户使用说明。

## 本轮不做

- 不为飞书编辑 / 撤回再造一套推测式兼容逻辑。
- 不把所有命令都改成可抢占 active run。
- 不引入多 session / 多实例级别的全局调度器。

这些仍然属于后续更大范围的交互与会话模型问题。
