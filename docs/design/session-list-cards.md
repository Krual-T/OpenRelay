# Session List Cards

更新时间：2026-03-10

## 目标

把 `/resume list` 从纯文本列表收敛为可交互卡片，并补上最小但稳定的翻页与排序能力。

本轮只解决三件事：
- 用户能直接在卡片里恢复会话
- 用户能在卡片里翻上一页 / 下一页
- 用户能在至少两种排序方式之间切换，且默认是“最近更新优先”

## 入口选择

本轮把 `/resume list` 作为第一条卡片化入口。

原因很简单：
- 它本来就是“浏览历史会话”的显式入口
- 不需要先改写 `/help` 或主回复卡片
- `/panel` 已经能跳到 `/resume list`，所以不必做双套入口

## 排序策略

当前提供两种排序：
- `updated-desc`：最近更新优先
- `active-first`：当前会话优先

默认使用 `updated-desc`。

这样做是为了让“浏览历史”这件事更符合直觉：
- 默认先看最近动过的会话
- 当前会话仍然保留高亮，但不强行置顶
- 如果用户明确更关心“当前会话 + 紧邻上下文”，再手动切到 `active-first`

## 分页策略

本轮采用固定页大小的简单分页，不引入游标。

原因：
- 当前数据来源仍然是本地会话 + 原生会话的合并列表
- 先把交互路径跑通，比过早引入复杂分页协议更重要
- 后续如果会话规模继续增长，再评估是否改成游标或更稳定的分页 token

## 按钮状态编码

卡片按钮不额外引入新的 webhook 状态字段，而是分两类编码：
- 翻页 / 排序动作直接编码进命令文本，例如 `/resume list --page 2 --sort updated-desc`
- 恢复动作保持最短主路径，直接使用 `/resume <session_id>`

这样做有三个好处：
- `parse_card_action_event` 不需要新增一层专用状态协议
- 卡片 action 进入 runtime 后，仍然表现为普通命令文本，主路径更短
- 浏览状态和恢复动作的语义分离：翻页 / 排序可追踪，恢复动作不携带多余状态噪音

## 模块落点

本轮边界如下：
- `SessionBrowser` 负责会话列表查询、排序和分页视图
- `SessionListCard` 负责把当前页渲染成飞书卡片
- `RuntimeCommandRouter` 负责解析 `/resume ... --page ... --sort ...`
- `AgentRuntime` 只负责发送卡片与失败回退

## 本轮不做

- 不把 `/panel` 全量改成独立分页浏览器
- 不引入复杂游标分页
- 不让排序状态写入持久化存储
- 不为卡片 action 再造一套独立于命令文本的状态机
