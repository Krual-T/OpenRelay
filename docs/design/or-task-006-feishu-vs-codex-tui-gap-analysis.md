# OR-TASK-006 Codex TUI 与 OpenRelay 飞书端体验差距调研

更新时间：2026-03-18

## 调研目标

这份调研不再回答“工程上还差哪些接口”，而是回答一个更直接的问题：

- 从飞书用户的日常使用体验看，`openrelay` 现在离“好用的远程 Codex 协作入口”还差什么？
- 哪些 Codex TUI 能力值得迁移到飞书？
- 哪些能力虽然 Codex 支持，但并不适合飞书场景，因此不应作为差距处理？

这里明确采用产品取舍视角，而不是 TUI 一比一对标视角。

## 判断原则

本调研统一按以下原则判断“是否算差距”：

1. **先看飞书场景是否真实需要**：群聊、线程、卡片交互、异步协作，和本地终端是不同场景。
2. **先看用户是否能顺畅完成任务**：比起命令是否齐全，更重要的是是否知道现在发生了什么、下一步该做什么。
3. **不追求命令面对齐**：Codex TUI 有的功能，不等于飞书一定要有。
4. **不为了技术可做而强行产品化**：如果一个能力只在本地终端自然，在飞书里会造成理解负担，就应该明确不支持。

## 先给结论

当前飞书端相对 Codex TUI 的主要问题，不是“少了多少命令”，而是“主流程还不够顺手”。

按用户体验优先级排序，当前差距应收敛为三类：

### P0：正在阻断主流程的体验缺口

1. **终端等待输入时，用户不知道怎么继续**
   - 飞书里现在可以看到系统在等输入，但还没有形成自然、稳定、低认知负担的回复方式。
   - 这是最接近“任务卡住”的问题。

2. **需要用户回答问题时，交互仍偏命令式，不够卡片化**
   - 用户输入、授权补充、MCP 提问这类场景，本质上是“填写/选择/确认”。
   - 如果还靠自由文本来回补，会让飞书体验更像调试接口，不像协作产品。

3. **用户很难一眼看懂当前会话正在做什么、停在哪里、下一步能做什么**
   - 飞书线程天然比终端更分散。
   - 如果缺少稳定的“当前状态 + 可执行动作”呈现，用户会频繁丢上下文。

### P1：影响效率，但不阻断任务

1. **当前会话的控制入口不够集中**
   - 停止、继续、压缩、切模型、查看状态等动作分散在不同命令和卡片里。
   - 用户需要记忆入口，而不是围绕“当前这轮会话”自然操作。

2. **历史会话与当前会话的关系不够清楚**
   - 虽然已经有恢复能力，但“我现在在哪个会话”“最近做过什么”“要不要继续这条线”仍不够直观。

3. **转录内容可看，但不够适合飞书里的异步回看**
   - 飞书用户更常见的动作不是持续盯着一块终端，而是隔一段时间回来快速扫一眼。
   - 因此“发生了什么、为什么停下、我该点哪里”比完整细节更重要。

### P2：可明确不追求对齐的差异

这部分不应继续作为“体验差距”施压：

1. **本地终端专属命令**
   - 如 `login`、`logout`、`cloud`、`features`、`completion` 等。
   - 这些更像本机环境或 CLI 自身管理能力，不适合投影成飞书主流程。

2. **终端原生交互手感**
   - 如键盘焦点、滚动、slash command 补全、alt-screen 连续操作。
   - 飞书不应模仿，也不可能模仿到位，应该改用卡片和线程语义重做。

3. **为了对齐而补齐低频命令**
   - 例如某些 CLI 子命令虽然存在，但若飞书用户缺乏明确场景，就不应先做。

## 以用户任务为中心重看差距

比起按后端事件或命令分类，更适合按飞书用户真实任务来判断。

### 场景一：我发起一个任务，希望它自己跑下去

当前体验基础已经具备：

- 可以发起任务
- 可以看到流式过程
- 可以在部分场景下做授权确认
- 可以中断当前运行

但体验还不完整：

- 遇到“需要我补一句输入”的时候会卡住
- 不容易一眼分辨“还在运行 / 在等我 / 已结束”
- 当前卡片对“下一步动作”的引导偏弱

结论：

- 主问题不是“能不能跑”，而是“跑到半路停住时，用户能不能自然接上”。

### 场景二：我不在现场，过一会回来看看发生了什么

这是飞书场景里非常核心、但终端场景没那么强的问题。

当前不足：

- 缺少强摘要视图，用户需要自己从转录里捞重点
- 缺少明确状态词：完成、失败、等待输入、等待授权、已停止
- 缺少稳定的“下一步建议”

结论：

- 飞书端应该优先把“异步回看可读性”做好，而不是优先补命令对齐。

### 场景三：系统问我问题，我想快速回答然后继续

这类场景包括：

- 终端等待输入
- user input
- MCP elicitation
- 需要用户补一个参数、说明、路径、选择项

当前不足：

- 回答方式还不够自然
- 结构化问题没有充分结构化呈现
- 用户不知道自己的回复是发到线程里，还是发回到某个等待中的交互

结论：

- 这类交互应该统一成“等待用户处理的卡片任务”，而不是分散成多种技术类型。

## 哪些 Codex 能力应迁移，哪些不应迁移

### 应迁移的是“能力目标”，不是命令形态

飞书端应承接的，不是 TUI 命令本身，而是以下用户目标：

- 启动和继续一条工作线
- 在运行中做必要确认和补充输入
- 清楚知道当前状态
- 在异步场景下快速回看和接续
- 围绕当前会话进行少量高频控制

### 不应迁移的是“只在本地终端自然成立的操作方式”

以下内容即使 Codex 支持，也不建议作为飞书对齐目标：

- 本地账户或 CLI 环境管理命令
- 纯终端手感型交互
- 需要频繁键盘微操作的流程
- 低频、低价值、只为“看起来像 TUI”而补的命令

## 建议的体验收敛方向

## 一、把“等待用户处理”统一成一种体验

不要再让用户区分：

- 这是 terminal interaction
- 那是 user_input
- 另一个是 MCP elicitation

从飞书用户视角，它们都是同一件事：

- **系统在等我做一个输入或选择**

建议统一成一类卡片：

- 明确说明现在在等什么
- 提供合适的输入控件或选项
- 提交后明确反馈“已发送，任务继续中”

这应是当前最高优先级。

## 二、把“当前会话控制”收敛到稳定入口

飞书用户不应该记很多命令。

更合适的模型是：

- 围绕当前运行中的会话，有一个稳定控制入口
- 常用动作直接就地可点
- 低频动作再下沉到命令或二级界面

高频控制只需要少数几个：

- 停止
- 查看当前状态
- 继续等待中的交互
- 压缩上下文
- 切换少量关键运行选项

## 三、把“异步回看”做成一等体验

飞书不是终端镜像，它首先是消息协作环境。

因此应优先强化：

- 当前状态标签
- 最近关键动作摘要
- 为什么停下
- 建议下一步
- 历史会话的快速预览和恢复

## 建议暂不追求的事项

以下内容建议在产品层面明确“暂不追求”，避免任务板继续把它们当缺陷：

- 完整复刻 Codex TUI 命令面
- 完整复刻 slash command 思维模型
- 把所有本地 CLI 能力都投影进飞书
- 用工程完成度替代用户体验完成度

## 最终结论

如果以飞书用户体验为中心，当前 `openrelay` 的关键问题可以收敛为一句话：

- **用户已经能在飞书里启动 Codex，但还不能始终顺畅地“看懂当前状态、接住等待中的交互、围绕当前会话继续推进”。**

因此后续优先级应调整为：

1. 先解决“等待用户处理”的统一交互体验。
2. 再解决“当前会话状态与控制入口”的集中呈现。
3. 最后再判断哪些 Codex 能力值得继续迁移。

而不是继续沿着“还缺哪些 TUI 命令 / runtime contract”展开。
