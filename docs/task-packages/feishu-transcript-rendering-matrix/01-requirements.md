# Requirements

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## Goal
建立一套可验证的飞书过程输出渲染矩阵：明确流式回复和最终 `Execution Log` 里可能出现哪些内容、每类内容应如何排版、哪些细节应该降噪或折叠，并通过 `lark-cli` 发送真实飞书卡片样例验证主要方案。

单一成功指标：本轮结束时，维护者可以拿着矩阵和真实飞书样例，判断每类过程输出在流式态与最终态应该用哪种渲染方式，并能解释为什么不会再在客户端暴露 `<font>`、`<text_tag>`、`&nbsp;`、`<br>` 这类原始标记。

## Problem Statement
目标用户是通过飞书使用 openrelay 的维护者和普通使用者。核心场景是：用户发送消息后，openrelay 在飞书卡片中流式展示执行过程，完成后再展示最终回复和可折叠的 `Execution Log`。

当前痛点有三个：

1. 过程输出复用了流式阶段的高亮信息，diff、命令输出、路径和状态文本会被渲染成 `<font>`、`<text_tag>`、`&nbsp;`、`<br>` 等飞书 Markdown 扩展标记；这些标记在最终 `Execution Log` 折叠面板里可能被客户端当作普通文本显示，导致截图中出现大段标签。
2. 过程输出信息量没有分层，`Updated files` 的 diff、测试命令输出、路径列表和状态摘要容易堆在同一个面板里，真正重要的最终回答反而被挤压。
3. `=====output=====` 这类内部调试分隔符可读性弱，既不像用户界面，也不能帮助快速区分命令、输出、diff、错误或验证结果。

现在要做这件事，是因为真实飞书截图已经证明当前渲染在客户端会错乱；继续局部替换某个分隔符或补几个换行，容易只修一个样例而遗漏其他输出类型。

## Required Outcomes
1. 输出类型矩阵
   - `acceptance criteria`：至少覆盖 command、web_search、reasoning、commentary、file_change、plan、collab、backend_event、summary、最终答案、运行中 spinner、错误输出、diff 输出、长输出这些类型，并标注它们来自哪条代码路径。
2. 渲染策略建议
   - `acceptance criteria`：分别说明流式态和最终态如何展示每类输出，包括是否显示详情、是否截断、是否允许颜色标记、是否使用折叠面板、是否保留原始输出。
3. 真实飞书卡片样例
   - `acceptance criteria`：用 `lark-cli` 发送一组覆盖关键类型的卡片样例，记录消息 id、发送命令、观察结果和已知盲点。
4. 实现边界
   - `acceptance criteria`：明确本轮后续如果进入实现，应改哪些模块、保留哪些兼容性、哪些输出暂时只做降级不做完整美化。

## Non-Goals
- 本轮不重新设计最终业务回答的文案风格；最终回答只需要不被过程日志干扰。
- 本轮不引入新的飞书长日志检索系统，也不把完整日志搬进数据库之外的新存储。
- 本轮不要求一次实现所有渲染改动；可以先形成矩阵、样例和可执行设计。
- `counterexample`：如果某个普通文本回复里出现 Markdown 列表缩进不理想，但不涉及流式过程输出或 `Execution Log`，它不属于本 task package。

## Constraints
- 真实飞书验证必须使用隔离的 `lark-cli` profile；除非用户明确要求，不使用 openrelay 服务应用作为 CLI 默认 profile。
- Python 相关命令统一使用 `uv run ...`。
- 飞书客户端的真实渲染效果优先于本地字符串单元测试；单元测试只能证明生成文本形态，不能替代真实卡片观察。
- 流式态和最终态可以采用不同渲染策略；不能因为流式态需要丰富提示，就把同样的高亮标记无条件带入最终 `Execution Log`。
- `cost cap`：本轮最多先发送一组小型真实飞书样例卡片，避免在用户会话里刷屏；大量矩阵验证应优先 dry run 或集中在少数综合卡片里。
