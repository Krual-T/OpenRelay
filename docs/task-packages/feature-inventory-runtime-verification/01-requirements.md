# Requirements

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## Goal
建立 openrelay 的功能清单与真实运行验证矩阵。做完这一轮后，维护者应能从一个清单里看到每个高频功能的用户入口、实际作用、状态副作用、可观测证据和验证方法。

单一成功指标：至少 10 个高频用户功能具备可执行验收用例，并且每个用例都能说明真实飞书触发后应该用什么本地证据判断通过或失败。

## Problem Statement
目标用户是 openrelay 维护者和使用飞书远程驱动 coding agent 的项目 owner。核心场景是：用户在飞书里发送消息、点击卡片或触发控制命令后，希望维护者能够判断“UI 上发生了什么、runtime 实际做了什么、backend 是否被调用、回复是否真正发出”。

当前问题是测试事实分散：

- `pytest` 能覆盖解析、路由、卡片结构和部分状态迁移，但不能证明真实飞书 UI 里发生了什么。
- `openrelay-trace` 已经能查询本地消息事件时间线，但还没有和功能清单、手动飞书验收用例形成稳定对应。
- 飞书官方 CLI 或官方调试工具可能能帮助观测消息、卡片、事件或流式 UI 表现，但它的能力边界尚未被官方资料和本项目实践确认。
- 近期已经出现过“UI 暴露已移除命令”“最终卡片漏掉 commentary”这类交互缺陷，说明只靠局部单元测试不足以支撑用户可见体验。

现在要做，是因为项目已经进入以真实飞书体验为主的阶段：继续新增功能前，需要先明确哪些功能存在、每个功能到底应该验证什么、哪些验证必须进入真实运行层。

## Required Outcomes
1. 建立功能清单初稿。
   - `acceptance criteria`：至少覆盖 `/help`、`/status`、`/resume`、`/resume latest`、`/workspace`、`/workspace select`、`/shortcut`、普通消息触发 backend turn、`/stop`、卡片按钮或表单动作。
   - 每项必须区分“用户看到的 UI”和“实际运行副作用”。
2. 建立验证矩阵初稿。
   - `acceptance criteria`：每个功能至少有一个验证方法，并标出它属于 `pytest`、本地模拟运行、真实飞书手动触发、trace 查询、官方工具观测中的哪一类。
   - 每个真实运行用例必须写出预期可观测证据，例如 trace stage、reply 记录、session 状态或 backend turn 事件。
3. 调研飞书官方 CLI 或官方调试工具的可用性。
   - `acceptance criteria`：明确它是否能发送测试消息、接收事件、查询消息、观测卡片更新或辅助判断流式 UI；不能确认的能力要记录为 unknown，而不是当成事实。
   - 调研结果必须附官方资料或真实命令证据。
4. 定义“用户手动触发，我来验证”的工作流。
   - `acceptance criteria`：给出至少一条端到端样例，说明用户在飞书发送什么，我如何通过 `openrelay-trace`、SQLite 或其他工具定位这条消息并判断结果。
5. 明确后续工具化方向。
   - `acceptance criteria`：写清楚是否需要新增 openrelay 自己的验证 CLI，或扩展 `openrelay-trace` 来承接功能用例验收。

## Non-Goals
- 本轮不实现完整自动化测试平台，不要求把所有真实飞书操作自动化。
- 本轮不承诺飞书官方 CLI 一定能验证 UI 流式表现；它只是必须调研的候选能力。
- 本轮不把所有低频配置项纳入清单，先聚焦高频用户入口。
- 本轮不替换现有 `pytest`，而是定义它和真实运行验证的边界。
- `counterexample`：为某个单独命令补一个单元测试，看起来也在“提升验证”，但如果没有进入功能清单和真实运行证据矩阵，就不属于本 task package 的完成结果。

## Constraints
- 本包必须使用 `task package` 命名和结构；旧称 `design package` 只能作为历史口径，不再作为当前事实源名称。
- 官方飞书能力必须通过官方文档、官方工具输出或真实运行证据确认；不能凭经验推断。
- 真实运行验证不能要求长期保存用户敏感消息正文；如果需要记录样例，应使用可公开的测试消息。
- `cost cap`：第一阶段只做清单、矩阵和官方工具调研，不超过一个小型文档与探针工具设计；如果需要实现完整 UI 自动化或官方工具封装，应拆出后续 task package。
- 不能破坏当前 `openrelay-trace`、SQLite 状态和已有 `pytest` 验证路径；新验证模型必须兼容现有可观测性基础。
