# Task Board Protocol

更新时间：2026-03-19

## 目标

这个协议不是为了引入额外系统，而是为了让 `Codex CLI` 直接通过修改仓库里的 Markdown 任务板，稳定地完成三件事：

- 找到当前任务的唯一入口
- 判断一个任务是否真的可以关闭
- 在完成实现后自己更新任务状态，而不是再靠人手工补记

目录职责约定：

- `docs/TaskBoard.md` 是任务面板的唯一事实来源。
- `docs/task-board-protocol.md` 负责定义任务板结构与更新规则。
- `docs/design/` 专门存放设计文档，不再承载任务面板或任务板协议。

## 基本方法

采用“**仓库内任务板 + 稳定任务 ID + 明确关闭条件 + 证据回写**”的低技术方案。

也就是说：
- 任务板文件就是事实来源。
- Codex 完成任务时，和代码改动放在同一轮 patch 里，同时更新任务板。
- 是否允许勾选，不靠主观描述，而靠任务条目里的关闭条件和证据字段。

## 任务条目结构

每个任务至少包含以下信息：

- **任务 ID**：例如 `OR-TASK-001`，用于稳定引用。
- **主复选框**：`[ ]` 表示未关闭，`[x]` 表示已关闭。
- **目标**：一句话说明为什么做这个任务。
- **当前关注**：说明当前仍未收敛的关键问题。
- **关闭条件**：只有全部满足，主复选框才能改成 `[x]`。
- **建议产物 / 已完成证据**：让 Codex 在修改后能回写文件路径，而不是只写一句“已完成”。
- **后续 follow-up**：任务主目标完成后，仍值得继续拆的小项。

## Codex 自勾选规则

Codex 可以自己勾选主复选框，但必须同时满足以下条件：

1. **任务 ID 明确匹配**：当前工作是针对某个明确任务，而不是“顺手修一点”。
2. **关闭条件已全部满足**：任务条目里的子项都能被当前改动证明。
3. **证据可回写到仓库**：至少能把设计 note、实现文件、测试文件或验证命令写回任务条目。
4. **状态与证据同一轮更新**：不能只改代码不改任务板，也不能只勾任务不补证据。
5. **如果只有部分完成，不勾主项**：只更新子项、证据、当前状态或 follow-up。

## Codex 完成任务时的操作顺序

建议固定为下面这五步：

1. 找到对应 `OR-TASK-xxx` 任务。
2. 先完成代码、设计文档或测试改动。
3. 运行该任务所需的最小验证。
4. 回写证据路径、已完成子项和 follow-up。
5. 只有在关闭条件全部满足时，才把主复选框从 `[ ]` 改成 `[x]`，并移到 `Landed / Follow-up`。

## 为什么这个方法适合 Codex CLI

因为 Codex CLI 天然擅长“同一轮同时修改多个文件”，而不擅长依赖外部任务系统状态。

这个方法的优点是：
- 不需要额外服务、数据库或 webhook。
- 任务状态和代码证据在同一个仓库里，回溯简单。
- 即使没有额外脚本，Codex 也能直接按文本规则执行。
- 后续如果要做自动校验，也可以基于这个结构补脚本，而不需要推翻现有文档。

## 不建议的做法

以下做法看起来省事，但长期不稳定：

- 只有一个“Open Questions”列表，没有关闭条件。
- 只写“已完成”，但不回写证据文件。
- 任务部分完成时直接勾选主项。
- 依赖会话记忆判断任务是否完成，而不更新仓库文档。
- 先引入复杂外部任务系统，再回过头同步仓库状态。

## 未来如果要再自动化

如果后续确实需要更强的自动化，可以在当前协议之上增加一个轻量脚本，做两件事：

- 校验任务条目是否缺少 `目标 / 关闭条件 / 证据` 等关键字段。
- 在 CI 或本地检查里阻止“主项已勾选但没有证据”的提交。

但在这之前，不建议先上脚本再反推文档结构；应该先把任务板结构稳定下来。
