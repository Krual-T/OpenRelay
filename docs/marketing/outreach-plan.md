# OpenRelay Outreach Plan

更新时间：2026-03-18

## 目标

先验证这件事是不是一个足够清晰、足够锋利的开源项目定位：

- 不是“飞书机器人”这个泛概念有没有人点开。
- 而是“把本机 coding agent 以真实 session 方式接进飞书”这个判断，能不能让目标用户立即理解价值。

## 三周节奏

### Week 1 - 打磨落点

- 清晰 README 首屏和项目定位。
- 准备一张社交分享图。
- 准备一段 60 到 90 秒演示录屏。
- 把首批对外文案统一成同一套说法。

关闭条件：一个第一次看到仓库的人，30 秒内能回答“它是什么”“它和普通 bot 的差别是什么”。

### Week 2 - 小范围验证

- 发 GitHub README 更新。
- 发 1 篇中文社区帖。
- 发 1 条英文社区帖。
- 把仓库发给 5 个已经在用 `Codex` 或飞书协作的人。

关闭条件：拿到至少 5 条具体反馈，且能分清是真需求、误解点还是噪音。

### Week 3 - 扩散与收敛

- 根据 Week 2 的误解点重写 README 前两屏。
- 把最有效的 demo 片段剪成更短版本。
- 再投 2 到 3 个更匹配的社区。
- 记录最常被问到的问题，更新 FAQ。

关闭条件：你能明确说出“谁最想要它、谁不该继续投”。

## 渠道策略

### 中文优先渠道

- 飞书 / AI / 独立开发者社群：最贴近使用场景。
- V2EX：适合验证标题是否足够锋利。
- 掘金：适合发完整架构与使用方式。

### 英文优先渠道

- X：测试一句话定位是否够强。
- Hacker News：测试“real session, not replayed history”这个判断能不能引起讨论。
- Reddit：收集不同工作流用户的反应。

## 每条内容都要回答的问题

- 它到底解决什么旧问题？
- 为什么不是另一个聊天壳？
- 为什么一定要强调真实 session？
- 为什么飞书 thread 是这个项目的自然宿主？

## 明确不做的事

- 不同时讲太多 backend 适配路线。
- 不把宣传重点放在“支持很多命令”。
- 不把帖子写成功能列表堆砌。
- 不过早承诺团队版、云服务版、托管版。

## 成功信号

下面这些反馈说明方向是对的：

- “我也不想迁移自己本机调好的 Codex 工作流。”
- “终于不是另一个把 session 假装连续的 bot。”
- “如果能在飞书 thread 里接着干活，这对我很有吸引力。”

下面这些反馈说明定位还不够清楚：

- “这和普通飞书机器人有什么区别？”
- “为什么不直接做成网页？”
- “是不是只是把历史消息重新发给模型？”
