# AGENTS.md

本文件定义 `openrelay` 仓库内的默认协作约定。除非用户在当前任务中明确覆盖，否则所有改动都应遵循以下原则。

## 1. 提交要求

- 每次完成一轮可独立成立的改动后，都应进行一次 `git commit`。
- 提交粒度应尽量聚焦：一个提交只解决一个明确问题，避免把不相关改动混在一起。
- 提交信息应准确描述“为什么改”以及“改了什么”，避免空泛标题。
- 如果任务天然包含多个独立部分，应拆成多个语义清晰的提交，而不是堆成一个大提交。
- 如果改动尚未通过最基本的自检，不应急于提交。

## 2. 工作气质

这个仓库偏好一种高度理性、结构化、结果导向的工作方式，可以理解为偏 `INTJ` 风格，但在工程语境下表达为以下要求：

- 先建立清晰模型，再开始实现；不靠堆试错推进复杂改动。
- 先判断长期正确性，再考虑局部便利；避免只解决眼前症状。
- 追求系统上的自洽、结构上的稳定、抽象上的克制。
- 偏好直接、明确、低噪音的表达，不喜欢模糊说法和表面包装。
- 允许必要的重构，但反对情绪化重写和无边界扩张。
- 更重视长期维护成本，而不是短期“先跑起来再说”。

## 3. 代码风格偏好

### 总体原则

- 偏好语义明确、层次分明、边界清楚的实现。
- 偏好高可读性，代码应让后来维护的人快速看懂。
- 偏好适度封装：让模块职责单一、接口稳定、调用点清晰。
- 偏好从整体结构出发组织代码，而不是靠零散补丁把逻辑拼起来。
- 不喜欢为了“看起来灵活”而引入过度抽象。
- 不喜欢冗余代码、重复分支、重复状态、重复包装。
- 不喜欢无意义的防御式兼容、历史包袱式保留、或“先留着以后也许有用”的残留代码。
- 不喜欢为了兼容旧实现而保留多余路径；如无明确要求，优先直接收敛到新实现。

### 具体要求

- 命名应直接表达真实语义，避免模糊缩写，避免靠上下文猜测含义。
- 一个函数只做一层抽象内的事情；过长时应按职责拆分。
- 模块之间的依赖关系应尽量单向、稳定，避免循环依赖和隐式耦合。
- 公共接口应尽量少而清楚，内部细节不要无意义外泄。
- 错误处理要明确，不要吞错，不要让失败路径变得不可见。
- 状态流转要可追踪，特别是异步、流式、事件驱动场景。
- 新增逻辑时优先整理结构，而不是继续叠加补丁式分支。
- 当局部实现开始破坏整体结构时，应优先调整结构，而不是继续迁就局部。

## 4. 兼容性策略

- 默认不为了“可能有人在用”而保留旧接口、旧分支、旧数据路径。
- 如果确实需要兼容，必须有明确理由，并控制兼容层范围。
- 能整体替换的地方，不要做长期双轨并存。
- 能删除的旧代码，不要保留“以后可能有用”的残留实现。
- 对兼容性的判断应基于真实需求，而不是想象中的风险。

## 5. 文档与结构要求

- 影响使用方式、配置方式、架构分层的改动，应同步更新文档。
- 新增设计决策、取舍理由、未决问题时，优先记录到 `docs/design/`。
- 如果一个文件正在变得难懂，优先通过重组结构改善可读性，而不是补注释掩盖问题。
- 如果一项设计需要靠大量口头解释才能成立，说明结构本身还不够清楚。

## 6. 实施方式

- 改动前先理解现有结构，再下手修改。
- 优先做根因修复，不做表面 patch。
- 尽量避免“顺手修很多地方”；除非确有必要，否则保持改动聚焦。
- 在不影响清晰度的前提下，保持实现简洁。
- 复杂改动应先想清楚边界、状态和数据流，再动手。
- 默认以“让系统更整洁、更稳定、更容易继续演化”为目标，而不只是“让这次需求通过”。

如果任务与上述原则冲突，以用户当前明确要求为准。
