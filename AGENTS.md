# AGENTS.md

本文件定义 `openrelay` 仓库内的默认协作约定。除非用户在当前任务中明确覆盖，否则所有改动都应遵循以下原则。

## 1. 提交要求

- 每次完成一轮可独立成立的改动后，都应进行一次 `git commit`。
- 提交粒度应尽量聚焦：一个提交只解决一个明确问题，避免把不相关改动混在一起。
- 提交信息应准确描述“为什么改”以及“改了什么”，避免空泛标题。
- 如果任务天然包含多个独立部分，应拆成多个语义清晰的提交，而不是堆成一个大提交。
- 如果改动尚未通过最基本的自检，不应急于提交。

## 2. 默认思维方式

这个仓库偏好一种高度理性、结构化、低噪音的工程风格。做改动时，默认应体现出以下倾向：

- 先理解整体，再改局部；先建立模型，再写实现。
- 先判断系统是否自洽，再考虑局部是否方便。
- 优先解决根因，而不是给症状打补丁。
- 重视长期稳定性、可维护性和演化能力，而不是只追求短期可用。
- 表达应直接、明确、克制，不堆术语，不做表面包装。
- 做决策时应有取舍意识：宁可少一点，也要清楚、稳定、可靠。

## 3. 代码风格偏好

### 总体原则

- 偏好语义明确、层次分明、边界清楚的实现。
- 偏好高可读性，代码应让后来维护的人快速看懂。
- 偏好适度封装：让模块职责单一、接口稳定、调用点清晰。
- 偏好从整体结构出发组织代码，而不是靠零散补丁把逻辑拼起来。
- 抽象必须服务于降低复杂度；如果抽象本身增加理解成本，就不应该存在。
- 不喜欢为了“看起来灵活”而引入过度抽象。
- 不喜欢冗余代码、重复分支、重复状态、重复包装。
- 不喜欢无意义的防御式兼容、历史包袱式保留、或“先留着以后也许有用”的残留代码。
- 不喜欢为了兼容旧实现而保留多余路径；如无明确要求，优先直接收敛到新实现。

### 具体要求

- 命名应直接表达真实语义，避免模糊缩写，避免靠上下文猜测含义。
- 一个函数只做一层抽象内的事情；过长时应按职责拆分。
- 模块之间的依赖关系应尽量单向、稳定，避免循环依赖和隐式耦合。
- 公共接口应尽量少而清楚，内部细节不要无意义外泄。
- 错误处理要明确，不要吞错，不要让失败路径变得不可见。
- 状态流转要可追踪，特别是异步、流式、事件驱动场景。
- 新增逻辑时优先整理结构，而不是继续叠加补丁式分支。
- 当局部实现开始破坏整体结构时，应优先调整结构，而不是继续迁就局部。
- 如果一个实现需要大量额外解释才能让人理解，通常说明结构还不够好。

## 4. 结构洁癖

- 主路径应尽量短、直、稳定，不应被边角兼容逻辑淹没。
- 相近逻辑应被收敛，而不是复制出多个相似版本。
- 同一层级的代码应保持相近的抽象粒度，不要忽粗忽细。
- 配置、状态、控制流、外部接口应各自归位，不要混成一团。
- 能通过重组结构解决的问题，不要优先靠注释、标志位或特判堆起来。
- 不要为了“保险”而无节制增加分支；每个分支都应有明确必要性。

## 5. 兼容性策略

- 默认不为了“可能有人在用”而保留旧接口、旧分支、旧数据路径。
- 如果确实需要兼容，必须有明确理由，并控制兼容层范围。
- 能整体替换的地方，不要做长期双轨并存。
- 能删除的旧代码，不要保留“以后可能有用”的残留实现。
- 对兼容性的判断应基于真实需求，而不是想象中的风险。

## 6. 文档与结构要求

- 影响使用方式、配置方式、架构分层的改动，应同步更新文档。
- 新增设计决策、取舍理由、未决问题时，优先记录到 `docs/design/`。
- 如果一个文件正在变得难懂，优先通过重组结构改善可读性，而不是补注释掩盖问题。
- 如果一项设计需要靠大量口头解释才能成立，说明结构本身还不够清楚。

## 7. 实施方式

- 改动前先理解现有结构，再下手修改。
- 优先做根因修复，不做表面 patch。
- 尽量避免“顺手修很多地方”；除非确有必要，否则保持改动聚焦。
- 在不影响清晰度的前提下，保持实现简洁。
- 复杂改动应先想清楚边界、状态和数据流，再动手。
- 默认以“让系统更整洁、更稳定、更容易继续演化”为目标，而不只是“让这次需求通过”。

## 8. Python / uv 约定

- 仓库内执行 Python 脚本、测试和维护命令时，默认使用 `uv run`，不要直接假设系统 Python 或当前 shell 环境可用。
- 如果某个本地脚本是仓库工作流的一部分，它依赖的 Python 包应加入本仓库的 `pyproject.toml`，而不是把“先手工装依赖”留在会话里临时说明。
- 只有在明确需要一次性临时依赖、且不值得进入项目环境时，才使用 `uv run --with ...`；这类情况应在对应文档或 skill 中明确写出。

## 9. 设计任务板约定

- `docs/design/TaskBoard.md` 现在作为设计与实现任务板使用，提交改动时应该看一下是否有本次完成的任务，并更新任务板。
- 新的设计 / 架构 / 体验类任务，优先以任务条目的形式进入任务板，而不是只补一段“问题描述”。
- 任务条目应尽量包含稳定任务 ID、目标、当前关注、关闭条件、建议产物或已完成证据、后续 follow-up。
- `docs/design/task-board-protocol.md` 是任务板详细协议；如果任务板条目需要收敛规则，应优先更新该文档，而不是在会话里临时约定。
- Codex 或其他协作者只有在关闭条件全部满足、且证据已在同一轮改动中回写到仓库时，才应把任务主复选框改成已完成。
- 如果任务只完成一部分，只更新子项、证据、状态或 follow-up，不要提前勾选任务主项。
- 已关闭任务应移动到任务板的已落地区域；未关闭但可继续推进的任务应保持在可执行区，而不是回退成松散备注。

如果任务与上述原则冲突，以用户当前明确要求为准。
